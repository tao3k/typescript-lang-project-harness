# CLI

The package exposes a thin command-line adapter over the default project
harness runner and the shared semantic-search protocol:

```shell
ts-harness search <workspace|prime|owner|dependency|deps|api|public-external-types|symbol|callsite|import|tests|text|ingest> ... [--json] [--package PATH] [PROJECT_ROOT]
ts-harness check [--changed | --full] [--json] [PROJECT_ROOT]
ts-harness agent doctor [--json] [PROJECT_ROOT]
ts-harness agent install --client codex [PROJECT_ROOT]
ts-harness agent guide --client codex [PROJECT_ROOT]
ts-harness agent hook --client codex <event> [PROJECT_ROOT]
```

When `PROJECT_ROOT` is omitted, the current working directory is used. Direct
agent-facing flows should use only `search`, `check`, `agent doctor`, or the
Codex hook lifecycle commands.
The public semantic-language identity is the registry entry advertised by
`ts-harness agent doctor --json`.

## Search Protocol

`search` is the primary read-only exploration surface:

```shell
ts-harness search workspace .
ts-harness search prime --package packages/core .
ts-harness search prime .
ts-harness search owner src/index.ts .
ts-harness search dependency react .
ts-harness search deps react/jsx-runtime@19.0.0::jsx .
ts-harness search api OrderStatus .
ts-harness search public-external-types react .
ts-harness search symbol OrderStatus .
ts-harness search callsite OrderStatus .
ts-harness search import ./order .
ts-harness search tests src/domain/order.ts .
ts-harness search text OrderStatus .
ts-harness search text OrderStatus owner tests .
rg -n "OrderStatus" src tests | ts-harness search ingest .
```

Compact text is the default agent-facing output. `--json` emits a structured
semantic-search packet with schema id
`agent.semantic-protocols.semantic-search-packet`, version `1`. The packet also
identifies the semantic language provider with
`protocolId=agent.semantic-protocols.semantic-language`,
`languageId=typescript`, `providerId=ts-harness`, `binary=ts-harness`,
`namespace=agent.semantic-protocols.languages.typescript.ts-harness`, and a
method such as `search/dependency`. The package carries the shared schemas in
`schemas/semantic-language-registry.v1.schema.json` and
`schemas/semantic-search-packet.v1.schema.json` so the TypeScript package CI can
validate protocol JSON from an independent checkout. When this package is
checked out as part of the protocol repository, the tests also compare those
package-local copies to the repository-level schema sources.

The first TypeScript implementation supports:

- `search workspace`: workspace package/router index with package edges
- `search prime`: bounded package-level owner/dependency/finding map plus
  TypeScript axes for `tsconfig`, known package extensions, build tools, and
  test/story/e2e surface; use `--package <path>` after `search workspace` to
  inspect one workspace package. `--view seeds` renders only a flow hint plus
  merged `|seed owner:` / `|seed symbol:` rows for first-turn agent routing.
- `search owner <path-or-owner>`: one owner plus adjacent semantic edges
- `search dependency <query>`: parser-owned `package.json` dependency facts plus
  TypeScript import-resolution usage for an external package
- `search deps <dep[/subpath][@version][::api]>`: version-aware dependency API
  usage with package manifest range, `currentWorkspaceVersion` when available,
  local import usage, and explicit `versionScope`.
  Local usage is attributed only when the requested version is the current
  workspace version; current usage packets use `versionScope=current`, and
  another requested version is marked `versionScope=external`.
- `search api <query>`: parser-owned exported/public API facts from the current
  workspace, including exported symbols, public function params, public data
  fields, and public async/effect return facts when the parser has them.
  Dependency-prefixed queries such as `react@18.0.0::jsx` are treated as
  external docs/API research and return an explicit provider-missing note rather
  than presenting current workspace parser facts as dependency-version docs.
- `search public-external-types <pkg>`: parser-owned public type surfaces that
  expose a dependency package. Direct `import("pkg").Type` text is reported as
  `public-external-type`; named imports such as `ReactNode` are reported as
  `possible-public-external-type` when the owner imports that dependency because
  named import binding attribution is not available yet.
- `search symbol <query>`: exported symbol definitions from parser-native
  export facts
- `search callsite <query>`: owner-level import/reexport sites for matching
  exported symbol owners
- `search import <query>`: import/reexport owner edges grouped by owner
- `search tests <owner-or-path>`: test owners that import a source owner
- `search text <query>`: owner-grouped path/export/source-text matches with
  compact hit evidence for `surface`, `ownerRole`, and bounded source snippets
- `search ingest`: stdin auto-detection for `rg -n`, vimgrep, path lists,
  diff paths, and `rg --json`

`search owner` resolves reasoning owners first, then parser-visible modules,
then existing project paths. If a parser-visible module is outside the
reasoning owner graph, the CLI returns `source=parser-visible-module`,
`parserOwner=false`, role/layer metadata, line counts, validity, and diagnostic
counts. If an existing path is outside the parser module set, it returns a
path-only owner with `source=path-only`, `parserOwner=false`, and
`next=ingest:<path>` rather than treating the path as a hard miss.

`search text` covers parser-visible source text, owner paths, and exports.
Docs, schema files, generated files, and other non-parser text should be
expanded with `rg` or `fd` and piped back through `search ingest`.
Compact `text` and `ingest` hit lines include filter evidence such as
`surface=source|test|workspace`, `ownerRole=...`, and `text=...` when a bounded
snippet is available. This gives agents non-query evidence for follow-up
filters, for example excluding tests or selecting definition-looking lines,
instead of piping the same search token back into `rg`.
Within equal relevance scores, text/import/symbol/callsite/ingest search ranks
newer project files before older paths, then falls back to path order. This
keeps recently edited code near the top without changing the packet schema.
`search api` is different from docs search: it returns bounded API shape facts
from parser-owned source, not prose. External dependency docs/API facts remain a
separate provider/source; `search deps <dep@version::api>` can point at
`search api <dep@version::api>`, but the API view must still preserve
provenance and emit `source=external-provider-missing` until that source exists.
`search public-external-types` shares the parser-owned API fact boundary. It is
useful for dependency leakage triage but is not a full declaration scanner yet:
it covers public function parameter types, async/effect surfaces, tuple
surfaces, and the available public data/type-boundary facts.
The first advertised multi-pipe form is final-only:
`search text <query> owner tests`. It keeps the text owners in the same packet
and appends test edges/hits for those owners instead of asking the agent to run
a separate `search owner` and `search tests` round.

## Public Repo Search Sandbox

The CLI flow is exercised against public TypeScript repositories to find
agent-search failure modes:

- Vite validates the workspace-first flow. `search workspace` must read
  `pnpm-workspace.yaml`, expose package `next=prime:<path>` targets, rank source
  workspace packages before `__tests__` fixture packages when the package list is
  capped, and keep `__tests__` / `.spec.` hits marked as `surface=test` so
  follow-up filters can exclude or select tests without searching the same token
  again.
- TypeScript validates large-repo robustness. Parser Debug Failure cases and
  deeply nested syntax must degrade to invalid modules or skipped native-syntax
  facts, not abort `search prime`. When compiler-program creation falls back to
  standalone parsing, import resolution still runs per file so prime keeps
  owner edges for later `owner`, `import`, and `tests` flow.

Project runs anchor at the nearest `package.json` above the requested path.
Running from a package subdirectory still evaluates the whole package project,
including `tsconfig`, package metadata, roots, modules, and edges relative to
that package root.

## Check Protocol

`check` runs the policy surface:

```shell
ts-harness check --changed .
ts-harness check --full .
ts-harness check --json .
```

`--changed` is currently a stable CLI lane that delegates to the project check;
future incremental narrowing can happen behind the same command.

For package validation, npm keeps implementation and policy lanes separate:
`npm run check:implementation` runs `tsc --noEmit`, while
`npm run check:policy` runs the repository self-policy gate. `npm test` is
composed from `test:implementation` and `test:policy` for the same reason.

Use `assertTypeScriptProjectHarnessEmbeddedClean()` when embedding the harness
inside `npm test` or `npm check`. It prints compact agent advice by default and
fails only on blocking findings. The embedded assertion defaults to a fast
non-semantic pass because `tsc --noEmit` or the test framework should own
type-check failure; callers can set `collectSemanticDiagnostics: true` when
they want the embedded assertion to collect TypeScript semantic diagnostic
advice too.

Use `assertTypeScriptProjectHarnessAgentClean()` only when agent-facing advice
should fail the test gate. The default `assertTypeScriptProjectHarnessClean()`
and CLI exit code remain blocking-only, so `info` advice can stay visible
without becoming a default policy failure.

## Registry Protocol

`agent doctor` advertises the TypeScript semantic language provider:

```shell
ts-harness agent doctor .
ts-harness agent doctor --json .
```

`agent install --client codex` writes `.codex/config.toml` for the current
repository. If the file already exists, the installer preserves other language
harness blocks and only updates or appends its marked `ts-harness` block. The
generated hooks delegate `SessionStart`, `UserPromptSubmit`, `PreToolUse`,
`PermissionRequest`, `PostToolUse`, `SubagentStart`, `SubagentStop`, and `Stop`
to the installed `ts-harness` binary:

```shell
ts-harness agent install --client codex .
ts-harness agent guide --client codex .
ts-harness agent hook --client codex pre-tool .
```

Run installation from the Codex session root that owns the conversation. In a
polyglot session rooted at another harness repository, install there so the root
project config concatenates the TypeScript block with the active language
harness block. The hook guide detects absolute TypeScript/JavaScript source
paths and points follow-up `ts-harness search` commands at the nearest
`package.json` or `tsconfig.json` project root.

The hook runtime consumes Codex hook JSON on stdin. It denies raw reads of
TypeScript/JavaScript source paths and broad source searches that have not
entered `ts-harness search ingest`, returning a compact `[ts-harness-flow]`
guide. The guide is a command-line command set made only of installed
`ts-harness` binary calls, with `|cmd` and `|pipe` lines for `prime`, `owner`,
`text owner tests`, `deps`, `ingest`, and `check --changed`.

The JSON form is a `semantic-language-registry.v1` document. The provider id
and binary are both `ts-harness`; the protocol identity remains
`agent.semantic-protocols.semantic-language`; the provider namespace is
`agent.semantic-protocols.languages.typescript.ts-harness`. The `methods` list
is the callable truth, and `methodDescriptors` records each method's command,
search view, output schemas, query/stdin/package-scope semantics, and
compact/JSON support.

Search descriptors also publish `capabilities` and optional
`ingestRequiredFor`. Each entry is a language-scoped descriptor:
`{languageId:"typescript", namespace:"semantic"|"typescript", name:<surface>}`.
The common semantic-language registry schema owns that shape only; the
TypeScript package owns the concrete vocabulary in
`schemas/typescript-semantic-capabilities.v1.schema.json`. The protocol
repository keeps a matching template at
`schemas/typescript-semantic-capabilities-template.v1.schema.json`; the template
is not the advertised provider schema, but the package test suite can compare
the local active vocabulary against it when this package is checked out as a
submodule.

`capabilities` names the surfaces the provider can search directly, such as
`parser-visible-source-text-search`, `dependency-api-token-usage-search`, or
`public-function-api-shape-search`. Public dependency leakage is advertised as
`public-external-type-search` plus `public-api-type-text-search`.
`ingestRequiredFor` names surfaces that must be expanded outside the provider
and normalized back through `search ingest`, such as docs text, schema JSON,
generated artifacts, non-parser paths, or external API docs.
`acceptedPipes` records final-only pipe names accepted after a query. In this
slice, `search/text` advertises `["owner","tests"]`.

## Reasoning Snapshot

When an agent needs the project reasoning tree before it chooses an edit
surface, use the protocol search surface:

```shell
ts-harness search prime --view seeds .
ts-harness search prime --package packages/core --view seeds .
```

The compact reasoning snapshot is not a separate public protocol command. It
follows the Rust harness agent snapshot shape internally: `Modules:`,
`OwnerBranches:`, `OwnerDependencies:`, and `FindingGroups:`. It includes module
roles, exports, package entry owners, package-name import owners,
project-reference/workspace owner provenance, and TypeScript-native import
resolution, including type-only edges, so repair agents can orient before
editing. Type-only edges come from native TypeScript syntax, including
declaration-level imports, specifier-level `type` markers, and `import("...")`
type queries. Rust-style source-shape counters such as `shadowed=` and
`orphaned=` are emitted only when non-zero. The format design, section order,
line shapes, and non-goals are defined in [Compact Agent
Snapshot](204_compact_agent_snapshot.md).
When root parser facts expose TypeScript project-reference packages or
workspace packages, the CLI renders each non-empty package scope with a short
`pkg <path>` heading. Each member package is run from its own `package.json`
anchor and local `tsconfig.json`; it does not inherit the root package config.
The single-package compact format is covered by
`tests/snapshots/agent_snapshot_project.snap`; the project-level workspace
format is covered by
`tests/snapshots/agent_snapshot_workspace_project.snap`.
When native TypeScript diagnostics exist, the snapshot groups their findings
under `FindingGroups:`. Full diagnostic cards, native TypeScript codes such as
`TS2322`, and related diagnostic information remain in the default compact
renderer and JSON output.
When TypeScript selects JavaScript via `allowJs`, JavaScript owners and edges are
rendered from the same parser-owned project report.

## Exit Codes

- `0`: no configured-blocking findings
- `1`: configured-blocking findings exist
- `2`: CLI argument error
- `3`: project-root, parser, or rendering error on protocol commands

The CLI delegates to library APIs. It does not own TypeScript parsing,
workspace discovery semantics, or project-specific policy.
